# CARD-FORMAT.md — 卡片：怎么写内容

> **更新频率**：格式规范确定后极少改动（高稳定性）
> **回答的问题**：写入记忆系统的内容，格式是什么？

---

## Zettelkasten 卡片格式（所有层级通用）

每张卡片必须**独立且完整**——单独阅读这张卡片即可理解全部内容，不依赖上下文。

```markdown
### [卡片标题：用一句话表达核心认知]

**是什么**
用2-4句话完整解释这个认知/行为/框架是什么。读者不需要了解对话背景就能理解。

**为什么重要**
为什么这条信息值得被记住？它解决了什么问题，或避免了什么失败？

**何时激活**
在什么情况下应该想起并使用这条认知？写3个具体的触发场景。

**具体例证**
用一个真实的例子说明。例子越具体越好，最好来自实际对话或项目。

**关联卡片**
→ [[其他卡片标题]]（说明关联关系）

**元数据**
- 来源：[用户原话 / Claude提炼 / 共同确认]
- 原文锚点：> "触发这条认知的原始引用"
- 首次日期：YYYY-MM-DD
- 确认次数：N
- 所属层级：L2/L3/L4 · 文件名.md
- 状态：[新认知·待验证] / [已验证] / [L4候选·N/3]
```

## 会话日报格式（ai-space/L1-memory/daily/）

```markdown
# 会话日报 YYYY-MM-DD

## 这次对话发生了什么
[完整叙述：用户带着什么需求，发生了什么关键转折，最终达成了什么。
要求：读这段文字的人即使没看过对话，也能完整理解发生了什么。]

---

## 今日沉淀的认知卡片

[直接嵌入完整卡片，使用上方的Zettelkasten格式。
**不要写"另见X文件"，直接把L1层的完整卡片写在这里。**
L2/L3层的卡片同时写入对应文件并在此说明：
"（另有N张卡片分别存于 domain-knowledge.md、mental-models.md 等，完整内容见对应文件。）"]

---

## 认知变化与冲突

[本次对话是否出现了与已有认知的冲突？
- 无冲突：写"本次对话与已有认知一致，无矛盾信号。"
- 有冲突：写明冲突内容、已有认知的来源、本次证据，不覆盖旧条目。]

---

## Skill 缺口候选

[是否发现了重复出现但未被抽象的工作流？
- 无：写"无。"
- 有：写明工作流描述 + 出现次数，**此处必须询问用户是否要创建新Skill**。]

---

## 权重变化

- 本次最活跃层：L?-?（N条新建/更新）
- 次活跃层：L?-?（N条新建/更新）
- 权重调整：参见 system/reading-weights.md 的更新记录
- 累计会话数：N
```

## 任务卡片格式（ai-space/tasks/）

> 设计原则：任务文件是多 Agent 协作的**中继站**。任何 Agent 随时可以读取它并完整理解上下文，随时可以追加执行日志，无需与之前的 Agent 沟通。

**任务 ID 命名规则**：`T-YYYYMMDD-NNN`（日期 + 当��序号，如 `T-20260327-001`）

```markdown
# T-YYYYMMDD-NNN [任务标题]

## 状态快照（可覆盖写入，保持最新）

- **状态**：未开始 / 进行中 / 已阻塞 / 待验收 / 已完成
- **创建时间**：YYYY-MM-DD Session N
- **最后执行**：YYYY-MM-DD Session N
- **下一步**：[下一个 Agent 应该做什么，越具体越好]
- **阻塞原因**：[如阻塞：描述什么条件满足后可以解除]

---

## 任务背景（不可修改，建立时写入）

**为什么这个任务存在？**
[用完整的叙事描述：这是哪个更大目标的一部分？用户原始需求是什么？
要求：任何没参与过之前对话的 Agent 读完这段，能完全理解为什么要做这件事。]

**上下游依赖**
- 依赖任务：#T-xxx（[已完成/进行中]）或"无"
- 被依赖任务：#T-xxx 或"无"
- 相关文件：[读这些文件可以补充背景]

---

## 任务目标（不可修改，建立时写入）

**交付标准**（完成的定义，可以打勾验收）
- [ ] [具体可验收的结果1]
- [ ] [具体可验收的结果2]

**范围内**：[明确做什么]
**范围外**：[明确不做什么，防止 Agent 过度执行]

---

## 前置条件（建立时写入，执行中可更新状态）

- [ ] [前置条件1]：[已满足/未满足，未满足时描述如何满足]
- [ ] [前置条件2]：[已满足/未满足]

---

## 执行日志（只追加，禁止修改之前的记录）

> 规则：每个 Agent 执行完毕必须追加一条日志。格式固定，方便下一个 Agent 快速扫描历史。

[YYYY-MM-DD Session N] **Agent角色/行动描述**
- 执行了：[做了什么，具体步骤]
- 产出了：[生成/修改了什么文件，或得到了什么结论]
- 发现了：[执行中遇到的意外情况或新信息]
- 遗留给下一个 Agent：[未完成的、需要继续的、需要注意的]

---

## 产出物（执行中持续更新）

| 产出物 | 路径/描述 | 状态 |
|--------|-----------|------|
| [名称] | [文件路径或描述] | [草稿/完成/废弃] |

---

## 交接备注（最后一个活跃 Agent 写入，供下一个 Agent 参考）

[如果你是下一个接手的 Agent，你需要知道：
- 最关键的上下文是什么
- 有哪些隐藏的约束或踩过的坑
- 为什么某些决定是这样做的（而不是其他方式）]
```

### 任务卡片写入规则

**何时创建任务卡片**：
- 任务明确需要跨会话追踪（本次对话完不成）
- 任务需要多个 Agent 分工协作
- 任务有明确的交付标准需要验收

**写入时机**（每个 Agent 必须遵守）：
1. **接手任务时**：先读任务卡片，理解完整背景再执行
2. **执行中遇到重要发现**：立即追加到执行日志（不要等到结束）
3. **执行完毕前**：更新「状态快照」+「交接备注」+「产出物」，追加执行日志条目
4. **任务完成时**：状态改为「已完成」，将文件从 `active/` 移动到 `completed/`

**禁止行为**：
- 禁止修改「任务背景」和「任务目标」（这是锚点，保证所有 Agent 的理解一致）
- 禁止修改之前 Agent 的执行日志条目（只追加，不覆盖历史）
- 禁止跳过「交接备注」（上一个 Agent 不写，下一个 Agent 就缺失关键上下文）

### ❌ 禁止：一句话卡片（信息不完整）

```markdown
### 用户喜欢零中断工作流
用户不喜欢AI在执行中途停下来问问题。
```

**问题**：三个月后看到这条，不知道为什么重要、什么时候用、有什么证据。

---

### ❌ 禁止：日报只写流水账

```markdown
## 今日沉淀
| 类型 | 内容 |
|------|------|
| 偏好 | 零中断 |
| 框架 | Piaget |
```

**问题**：表格记录的是"发生了什么事"，而不是"沉淀了什么可复用的认知"。

---

### ❌ 禁止：卡片互相依赖、不能独立阅读

```markdown
### 阻塞点优先于执行步骤

见上文的「宪法」设计原则，这是其延伸。
```

**问题**：单独读这张卡片什么也不知道。每张卡片必须自解释。

---

### ❌ 禁止：来源标注缺失

```markdown
**元数据**
- 首次日期：2026-03-27
```

**问题**：缺少 `来源` 字段。无法判断这是用户说的还是AI编的，违反宪法规则C4。

---

### ✅ 正确：完整、独立、自解释的卡片

见本文件顶部的卡片格式模板。每个字段都有存在的理由，不能省略。
